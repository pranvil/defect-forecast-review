# 缺陷预测算法说明

本文档说明当前“新项目预测”的预测算法口径和实现流程。前端 mock 与后端导出服务保持同一套核心逻辑；生产接口以后端 `export-service/app/logic.py` 为准，前端本地演示逻辑主要在 `defect-forecast-web/src/utils/defectCalculation.ts` 和 `defect-forecast-web/src/utils/forecastDistribution.ts`。

## 1. 输入数据

预测输入由以下几类数据组成：

1. 项目基础信息：项目名称、开始周期、结束周期、项目类别、目标区域、操作系统、设备类型等。
2. 技术变量：芯片平台、新旧、流水线、外包商、需求量、投入人力、是否支持 SIM、运营商、用户测试等。
3. 参考项目：自动识别的 Top 3 相似历史项目，以及用户手工添加的参考项目。
4. 节点信息：节点名、周期、日期、开发解决率、测试完成率、测试提交率。
5. 当前项目团队选择：测试团队和开发团队，默认全选，可由用户取消勾选。
6. 系统配置里的聚合团队：用于把历史项目中的原始团队名映射到当前聚合团队。

## 2. 总 Defect 数量预测

总量预测先从历史项目中找相似项目，再基于项目变量做系数修正。

### 2.1 相似项目筛选

系统会计算新项目与历史项目的相似度。普通项目使用以下权重：

| 字段 | 权重 |
| --- | ---: |
| 项目类别 | 0.25 |
| 区域 | 0.20 |
| 操作系统 | 0.15 |
| 芯片平台 | 0.10 |
| 运营商重叠度 | 0.20 |
| 需求量区间 | 0.10 |

IDH 项目使用另一组权重：

| 字段 | 权重 |
| --- | ---: |
| 项目类别 | 0.20 |
| 外包商 | 0.25 |
| 区域 | 0.15 |
| 操作系统 | 0.10 |
| 芯片平台 | 0.10 |
| 运营商重叠度 | 0.15 |
| 需求量区间 | 0.05 |

过滤规则：

- 设备类型不一致时，历史项目不参与相似度候选。
- IDH 与非 IDH 项目不混用。
- 只保留 defect 总数有效且大于 0 的历史项目。
- 默认取相似度最高的 Top 3；如果用户手工添加了参考项目，后续 team 占比也会参考这些项目。

### 2.2 基准总量

基准值为 Top 3 相似项目 defect 总量的简单平均值：

```text
baseValue = avg(top3.defects)
```

### 2.3 修正系数

最终总量按以下系数修正：

| 系数 | 规则 |
| --- | --- |
| 芯片新旧 | New 为 0.2，否则 0 |
| 运营商 | `(运营商数量 - 1) * 0.1` |
| 用户测试 | 0 个为 0；1 个为 0.1；每多 1 个增加 0.05，最多增加到 0.25 |
| SIM | 不支持 SIM 为 -0.2，支持为 0 |
| 投入人力 MM | 与参考项目平均 MM 对比，增加值最多 0.2 |
| 流水线 | 全部 0.08；冒烟 0.01；无冒烟 -0.01；不部署/无 -0.08 |

最终公式：

```text
estimatedDefects =
  round(baseValue
    * max(0.1, 1
      + chipset
      + operators
      + userPrograms
      + supportSim
      + mm
      + pipeline))
```

如果找不到相似项目，前端 mock 会退回使用基础周数据合计；后端预测依赖历史库可用数据。

## 3. 预测周期范围

预测周期从项目基础信息的开始周期开始。

结束周期取以下两者中更晚的一个：

1. 项目基础信息的结束周期。
2. 节点信息中最后一个填写了周期的节点。

这样可以避免项目结束时间早于后续节点时，把剩余 defect 强行压缩到项目结束周。

如果第一个有周期的节点与项目开始周期不一致，前端会阻止预测，并要求用户修改项目开始时间或节点周期。

## 4. 节点指标口径

节点信息中三个指标含义如下：

| 指标 | 口径 | 是否参与预测计算 |
| --- | --- | --- |
| 测试提交率 | 累计提交率，即当前累计 created / 最终预测 defect 总量 | 是 |
| 开发解决率 | 当前累计 fixed / 当前累计 created | 是 |
| 测试完成率 | 测试完成考核展示指标 | 否，只展示在 Excel/预览 |

重要口径：

- 节点周期表示准入节点所在周。
- Excel 预览和导出中，节点名与三个指标都显示在节点自己的周期。
- 预测计算时，测试提交率和开发解决率使用节点前一周的累计数据作为准入要求。
- 如果节点已经是预测范围第一周，没有前一周，则使用第一周。

例如：

```text
M1-4 在 26W21
测试提交率 = 30%
开发解决率 = 50%
测试完成率 = 25%
```

展示：

```text
MV 版本、问题提交率、问题解决率、测试完成率都填在 26W21
```

计算：

```text
26W20 的累计 created 需要约束到最终预测总量的 30%
26W20 的累计 fixed 需要约束到 26W20 累计 created 的 50%
```

## 5. Created 每周分布

Created 每周提报数量优先呈现钟形分布。

### 5.1 从节点提交率推断峰值

系统先收集所有填写了周期和测试提交率的节点，形成约束点：

```text
constraint.index = 节点周期的前一周
constraint.rate = 测试提交率
```

随后使用正态分布的反函数推断峰值位置：

```text
rate = clamp(rate, 1, 99)
peakIndex ≈ constraint.index - sigma * inverseNormalCdf(rate / 100)
sigma = max(1.5, 周数 / 5)
```

如果有多个约束点，取多个峰值估计的平均值。

如果没有提交率约束，峰值默认在周期中间。

### 5.2 生成钟形权重

根据峰值位置生成每周权重：

```text
weight[i] = exp(-0.5 * ((i - peakIndex) / sigma)^2)
```

### 5.3 按约束分段分配

系统把所有测试提交率约束转换成累计 created 目标：

```text
targetCumCreated = round(estimatedDefects * testSubmissionRate / 100)
```

同时最终周强制加入 100% 目标：

```text
finalCumCreated = estimatedDefects
```

然后按约束点分段，把每段增量按钟形权重分配到各周。

分配后会做整数化处理，保证每段整数求和等于该段目标增量，最终累计 created 等于预测总 defect 数量。由于强行分段找齐目标，部分节点前后可能会出现曲线不平滑（锯齿形突变）的现象，但优先保证了节点约束的绝对满足。

## 6. Fixed 每周分布

Fixed 最终总量必须等于预测 defect 总量，但不强制呈现 created 一样的钟形。

### 6.1 开发解决率约束

开发解决率不是按最终 defect 总量做分母，而是按当前累计 created 做分母。

对每个填写了开发解决率的节点：

```text
constraint.index = 节点周期的前一周
targetCumFixed = round(cumCreated[constraint.index] * devResolutionRate / 100)
```

例如：

```text
M1-4 的开发解决率 50%
若 M1-4 前一周累计 created = 300
则该约束要求累计 fixed ≈ 150
```

### 6.2 Fixed 权重

Fixed 权重基于前面 created 的历史窗口生成，倾向于滞后于 created：

```text
fixedWeight[i] =
  created[i - 1]
  + created[i - 2] * 0.8
  + 1
```

也就是说，某一周的 fixed 更受前一周和前两周 created 影响。

### 6.3 Backlog 保护

只要保证 backlog 为非负数（即累计 fixed 不超过累计 created），且在最终周收敛到 0 即可。系统不再强制用正弦曲线保留多余的缓冲池，完全尊重用户设定的各节点解决率目标。

中间周的 fixed 累计上限为：

```text
maxCumFixed[i] = cumCreated[i]
```

最终周不保留缓冲：

```text
finalCumFixed = finalCumCreated = estimatedDefects
```

如果某个开发解决率要求会让 backlog 过低，系统会按上限计算，并给出提醒。

### 6.4 Fixed 可用性校验

Fixed 不允许超过已经 created 的数量。系统会逐周校验：

```text
available = cumCreated[i] - cumFixedBeforeThisWeek
fixed[i] = min(plannedFixed[i] + carry, available)
```

无法在当前周消化的 fixed 会结转到后续周。

## 7. Backlog 计算

Backlog 按累计 created 与累计 fixed 差值计算：

```text
backlog[i] = cumCreated[i] - cumFixed[i]
```

当前设计目标：

- backlog 不允许为负。
- backlog 不要求每周为 0。
- backlog 可以上升和下降，优先保证 created 分布接近钟形。
- 最终周 fixed 总量等于 created 总量，因此最终 backlog 为 0。

## 8. 冲突提醒

系统会检查节点指标与预测曲线的偏离程度。

阈值：

```text
12 个百分点
```

如果节点指标与当前钟形分布推导出的建议指标偏差超过 12 个百分点，会生成提醒。

提醒会同时给出两类建议：

1. 调整指标值。
2. 调整节点周期。

示例：

```text
M4 的测试提交率 85% 与钟形分布偏离 18 个百分点，
建议指标调整到 67% 或将节点周期调整到 26W33。
```

## 9. Team 拆分

Team 拆分发生在总 created/fixed 周分布之后。

### 9.1 使用当前勾选团队

新项目预测页面里，测试团队和开发团队默认全选。用户可以取消勾选不参与当前新项目的团队。

注意：

- 勾选项表示新项目要分配到哪些 team。
- 勾选项不用于过滤历史项目。

### 9.2 历史占比来源

team 占比使用当前参考项目：

- 系统自动识别的 Top 3。
- 用户手工添加的更多参考项目。

### 9.3 聚合团队映射

历史项目中可能是原始 Jira 团队名，而预测输出使用系统配置里的聚合团队。

映射规则：

- 优先匹配聚合团队名称。
- 其次匹配聚合团队备注中的别名。
- 支持逗号、中文逗号、分号、中文分号分隔。
- 也会尝试后缀匹配和包含匹配。

### 9.4 Created 与 Fixed 分别分配

Created 分配到测试团队：

```text
测试团队占比 = 历史参考项目中该聚合测试团队 created 数量 / 所有勾选测试团队 created 数量
```

Fixed 分配到开发团队：

```text
开发团队占比 = 历史参考项目中该聚合开发团队 fixed 数量 / 所有勾选开发团队 fixed 数量
```

每周分配时按比例拆分，并做整数化处理，保证各 team 当周合计等于该周总 created/fixed。

### 9.5 缺失报错阻断

如果新勾选的团队在当前参考项目（历史数据）中没有可用的 team 历史占比：

- 系统会**抛出异常报错并阻断预测**。
- 提示用户：“勾选的团队 [X] 在历史参考项目中没有占比数据，请手动输入预估占比后再进行预测。”

## 10. Excel 预览与真实导出

### 10.1 前端预览

“Excel 模板预览”不是直接打开真实 `.xlsx` 模板，而是前端按模板结构渲染的 HTML 表格。

预览数据来自当前预测结果 dataset，并会用当前“节点信息”补齐 milestones，确保节点指标能实时显示。

### 10.2 真实导出

真实导出使用模板：

```text
export-service/templates/DRP_template.xlsx
```

导出时后端会：

1. 加载模板。
2. 清理模板中的历史外部链接和失效命名区域，避免 Excel 打开时提示修复。
3. 清空模板中 `C:AB` 的周数据区域。
4. 重新写入 Month、Date、Week。
5. 动态调整 Created/Fixed team 明细行。
6. 写入 Total、累计、Backlog、MV 版本和三项节点指标。

Month 行会按本次预测周日期重新合并，不再保留模板原始月份。

### 10.3 节点指标展示位置

Excel 中：

- `MV 版本` 显示节点名，落在节点自己的周期。
- `问题提交率` 显示测试提交率，落在节点自己的周期。
- `问题解决率` 显示开发解决率，落在节点自己的周期。
- `测试完成率` 显示测试完成率，落在节点自己的周期。

这只是展示位置；预测计算仍使用节点前一周的累计数据作为准入约束。

## 11. 结果保存

“新项目预测”页面点击保存后，保存当前 input 与预测 result。

“预测结果”页面只展示已经保存过的预测结果，并支持删除保存版本。

保存的 result 中包含：

- weekly created/fixed/backlog。
- createdTeams。
- fixedTeams。
- milestones。
- 总量预测值。
- 参考项目。
- 修正系数。
- warning 提醒。

## 12. 当前实现边界

当前算法是规则型预测，不是训练型机器学习模型。

主要假设：

- 总量来自历史相似项目均值与业务系数修正。
- created 周分布尽量符合钟形。
- fixed 滞后于 created，同时满足开发解决率准入和最终 fixed 总量。
- backlog 不允许为负，最终回到 0。
- team 拆分使用历史占比，不再单独改变 created/fixed 总量。

后续如果要继续演进，建议优先明确以下方向：

- 是否允许最终 backlog 不为 0。
- 是否需要使用更多历史项目分布学习峰值宽度。
- fixed 是否需要引入团队产能上限。
- 冲突阈值 12 个百分点是否需要配置化。
- team 历史占比是否需要按项目类型、区域或周期阶段加权。
