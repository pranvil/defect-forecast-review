# 手机项目缺陷预测与对比分析工具 - 需求文档 v2

## 1. 项目目标
开发一个可在本地独立使用的工具，用于：
- 管理历史项目数据
- 输入新项目信息后自动匹配相似历史项目
- 基于历史项目估算新项目总缺陷量和各 team 总量
- 根据项目周期与版本节点考核要求生成每周分布
- 对历史项目、预测结果、JIRA 实际数据进行查看与对比
- 导出可直接使用的 Excel 表格
- 支持多人各自在本机使用，并通过导入/导出共享历史项目数据

## 2. 产品定位
这是一个本地 Web UI + 本地数据库 + 安装包分发的项目分析工具，不是在线平台，也不是训练型 AI 系统。

当前阶段不做：
- AI 修正
- 模型训练
- 中央共享数据库
- 多人实时在线协作
- 多项目复杂联动分析
- team 曲线图

## 3. 核心业务逻辑

### 3.1 总量预测层
基于历史项目相似度估算：
- total created
- total fixed
- 各 team total created / fixed

总量预测时不考虑节点信息。

总量预测主要考虑：
- 产品类型
- 是否支持 SIM
- 平台
- 芯片型号（辅助）
- OS 版本
- 是否大版本升级
- 项目角色（leading / variant / refresh）
- 项目周期时长
- 是否引入 end user
- team 规模
- 项目能力信息

### 3.2 周分布生成层
根据以下信息生成每周分布：
- 项目起始时间
- 真实周规则
- 项目周期
- 节点时间
- 节点考核标准

输出：
- weekly created
- weekly fixed
- cumulative created
- cumulative fixed
- backlog

结论：
历史项目相似度只负责估算总量；具体每周分布由项目周期与节点考核要求决定。

补充说明（总量修正规则口径）：

- 总量预测的基准值来自参考项目的 defect 总量（如 Top 相似项目的均值）。
- 业务变量修正采用**相对参考项目的差异修正**：当新项目与参考项目在某项变量一致时，该变量不会带来总量变化；只有当变量与参考项目存在差异时，才会按规则对总量进行上调/下调。

## 4. 周规则定义
系统使用真实自然周，规则如下：
- 每年第一周从 1 月 1 日开始，到当周周日结束
- 后续每周为周一到周日
- 周编号按自然年计算
- 所有历史数据、预测数据、JIRA 聚合统一使用这套规则

例如：
- 2026-01-01 是周四
- 2026 年第 1 周为：2026-01-01 ~ 2026-01-04

## 5. 使用场景

### 5.1 历史项目录入
用户录入一个已结束或可归档的项目：
- 手工录入
- 或从 JIRA 初始化导入

系统生成：
- team 汇总
- 每周 created/fixed/backlog 数据
- 项目摘要信息

用户再补充：
- 项目角色
- 产品信息
- 能力信息
- end user 信息
- 节点信息（可选）

最后保存到数据库。

### 5.2 新项目预测
用户输入新项目基础信息后，系统：
- 自动匹配相似项目
- 推荐参考项目
- 允许人工增删参考项目
- 估算总量
- 根据节点要求生成周分布
- 允许手工微调
- 保存预测版本
- 导出 Excel

### 5.3 单项目查看
如果用户只选择一个项目，系统也要完整展示该项目的信息，包括：
- 项目基础信息
- 汇总指标
- team 数据
- 周分布表
- 曲线图
- 节点信息（如有）

### 5.4 双项目对比
支持：
- 历史 vs 历史
- 预测 vs 历史
- 预测 vs JIRA 实际

展示：
- 基础信息对比
- 汇总指标对比
- team 数据对比
- 周总量对比
- 曲线图对比
- 节点信息显示

### 5.5 JIRA 实际偏差监控
对进行中的项目：
- 从 JIRA 拉取实际数据
- 与预测结果对比
- 判断偏差
- 输出预警提示

## 6. 功能需求

### 6.1 team 配置管理
team 不能写死在代码中，必须通过 UI 配置。

支持：
- 测试 team
- 开发 team
- 通用 team

每个 team 至少支持：
- team 名称
- team 类型
- 是否启用
- 排序
- 别名映射

别名映射用于处理：
- 历史数据中的名称差异
- JIRA 字段中的 team 名称差异

### 6.2 项目能力配置管理
项目能力采用可扩展配置，不写死字段。

支持在 UI 中维护能力字典，例如：
- WFC
- VoLTE
- VoNR
- E911
- MMS
- eSIM
- DSDS
- Camera
- BT
- Wi-Fi
- DPC
- SUW
- OTA
- VVM
- RCS
等

每个能力支持：
- 能力名称
- 分类
- 默认风险等级
- 是否启用

项目中记录能力时支持：
- 是否支持
- 风险等级
- 是否新增能力
- 是否有 operator 定制

### 6.3 历史项目管理
#### 6.3.1 支持新增
历史项目支持新增。

#### 6.3.2 支持修改
历史项目允许修改，用于修正录入错误或补充信息。建议记录最后更新时间，后续可扩展修改日志。

#### 6.3.3 支持 JIRA 初始化导入
导入流程：
1. 输入 JIRA 查询条件与项目统计周期
2. 拉取 issue 数据
3. 自动聚合：
   - 周总量
   - 各 team 周分布
   - 各 team 总量
4. 用户人工补充业务字段
5. 保存为历史项目

#### 6.3.4 历史项目节点信息
历史项目可以保存节点信息，但不是强制项。没有节点信息的老项目仍可正常入库和使用。

### 6.4 历史数据共享
由于没有中央共享环境，采用：
- 本地数据库
- 导入/导出项目包

支持：
- 导出单个历史项目
- 导出多个历史项目
- 导出完整历史项目包
- 导入历史项目包

导入时支持：
- 跳过已存在项目
- 另存为副本
- 覆盖（可选）

### 6.5 新项目预测
#### 6.5.1 新项目信息输入
至少包括：
- 项目名称
- 产品类型：smartphone / tablet / feature phone
- SIM 类型：full sim / data only / no sim
- 平台：MTK / QCOM / other
- 芯片型号（可选/辅助）
- OS 版本
- 是否大版本升级
- 项目角色：leading / variant / refresh
- 复用来源项目（可选）
- 项目起始时间
- 项目周期（周）
- 是否引入 end user
- end user 规模
- 测试 team / 开发 team 规划（可选）
- 项目能力信息
- 节点信息
- 节点考核标准

#### 6.5.2 自动匹配相似项目
系统自动推荐相似历史项目，并展示：
- 项目名称
- 相似度分数
- 产品类型
- SIM 类型
- 平台
- OS
- 角色
- 项目周期
- total created
- total fixed
- end user 信息

#### 6.5.3 人工调整参考项目
用户可以：
- 删除推荐项目
- 手工添加历史项目
- 确定最终参考项目集

当前版本先不做复杂权重编辑器。

#### 6.5.4 预测结果生成
系统输出：
- total created
- total fixed
- 各 team total created / fixed
- weekly created
- weekly fixed
- cumulative created
- cumulative fixed
- backlog

#### 6.5.5 预测结果手工微调
预测结果生成后允许人工调整。

支持调整：
- total created
- total fixed
- 某个 team 的 total created / fixed
- 每周 created / fixed
- 节点目标值

调整后系统自动重算：
- cumulative created
- cumulative fixed
- backlog

team 自动调整规则：
- 当用户修改某个 team 总数时，提供选项：是否自动调整其他 team 占比
- 默认：不自动调整
- 如果勾选自动调整，则其余 team 按原比例缩放

#### 6.5.6 预测版本管理
同一项目支持保存多个预测版本。每个预测版本支持：
- 保存
- 查看
- 删除

删除规则：
- 默认软删除
- 设置中支持永久清理

### 6.6 节点体系与考核标准
#### 6.6.1 节点类型
支持：
- M1-x（如 M1-1, M1-2）
- M2
- M3
- M4
- M5
- V1
- V2
- V3
- V4

规则：
- M1 可以有多个子节点
- M2-M5 不一定全部存在
- V 节点最少 1 个，最多 4 个
- 一个项目总节点数可能从几个到十几个不等

#### 6.6.2 节点字段
每个节点至少包括：
- 节点名称
- 节点日期
- 节点类型
- 节点顺序
- 是否考核节点
- 测试提交率目标
- 开发解决率目标
- 备注

#### 6.6.3 指标定义
测试提交率定义为：
累计 created / 预测 total created

开发解决率定义为：
累计 fixed / 累计 created

#### 6.6.4 节点目标性质
节点目标是软约束，不是绝对硬约束。

如果节点目标导致：
- 分布不自然
- 峰值异常
- backlog 畸形
- 曲线明显不合理

系统需：
- 给出警告
- 提示用户是否修改考核指标
- 推荐“当前节点 + 后续节点”的一组平滑目标值

补充：分布形态的优先级

- Created 不再强行要求钟形；优先满足节点的提报率 / 解决率等指标约束。
- Backlog 形态优先呈现单峰（近似钟形）并在尾部进入收敛阶段；若与节点约束冲突，应优先满足节点约束，并给出提醒。

#### 6.6.5 项目结束状态
项目结束时允许：
- total fixed < total created
- backlog > 0

不要求项目尾部必须收敛到 0。

补充约束：

- 最终 backlog 允许 >0，但必须极少（建议默认阈值：`max(5, round(totalCreated * 0.005))`）。

### 6.7 对比分析
#### 6.7.1 单项目查看模式
如果用户只选 1 个项目，仍应显示：
- 项目基础信息
- 汇总指标
- team 数据
- 周分布表
- 曲线图
- 节点信息

#### 6.7.2 双项目对比模式
支持：
- 历史 vs 历史
- 预测 vs 历史
- 预测 vs JIRA 实际

展示：
- 基础信息对比
- 汇总指标对比
- team 数据对比
- 周总量对比
- 曲线图叠加
- 节点信息显示

#### 6.7.3 team 展示规则
team 只做表格，不做曲线图。

#### 6.7.4 周总量展示规则
周总量需要：
- 表格
- 曲线图

至少展示：
- weekly created
- weekly fixed
- cumulative created
- cumulative fixed
- backlog

#### 6.7.5 节点在图中的展示方式
图的横坐标是周信息。若某一周对应节点，则在该周下显示节点信息。
例如：
- W25 对应 M5
- 则在 W25 下方显示 “M5”

单项目模式和双项目模式都应支持显示节点信息。

### 6.8 JIRA 实际数据拉取与预警
#### 6.8.1 JIRA 字段映射
需支持配置：
- created date 字段
- fixed 判定字段
- fixed 判定值
- team 字段
- status 字段
- priority / severity 字段

#### 6.8.2 Fixed 判定规则
当前默认：
- Verified_sw 算 fixed

但后续字段可能变化，系统必须支持通过 UI 配置：
- fixed 判定字段
- fixed 判定值

不能写死在代码里。

#### 6.8.3 历史项目导入统计周期
从 JIRA 导入历史项目时，由用户输入统计起始时间/统计周期。因为有些项目结束后仍在维护，这些后续 bug 不纳入参考周期。

#### 6.8.4 实际数据聚合
系统按统一周规则聚合：
- weekly created
- weekly fixed
- cumulative created
- cumulative fixed
- backlog
- 各 team total / weekly 数据

#### 6.8.5 偏差分析
系统对比：
- 预测 vs 实际累计 created
- 预测 vs 实际累计 fixed
- 预测 vs 实际 backlog
- team 偏差

#### 6.8.6 预警规则
第一版采用规则型预警，例如：
- 实际累计 created 高于预测一定比例
- 实际累计 fixed 低于预测一定比例
- 实际 backlog 高于预测一定比例
- 进入预期收敛期但实际 created 仍偏高
- 某个 team 偏差明显

### 6.9 Excel 导出
支持导出：
- 预测结果
- 单项目详情
- 双项目对比结果

导出内容至少包括：
- 项目摘要
- team 数据
- 每周 created/fixed/backlog
- 节点信息
- 关键指标

## 7. 技术路线
### 7.1 总体技术路线
采用：
- 本地 Web UI
- 本地数据库
- 安装包分发
- 本地运行，不依赖中央服务器

### 7.2 推荐技术栈
后端：
- Python
- FastAPI

前端：
- React
- Ant Design
- ECharts

数据库：
- SQLite

Excel：
- openpyxl

安装包：
- Windows 安装包
- 内置 Python 运行环境
- 用户无需自己安装 Python

### 7.3 升级与数据保留
采用：
- 安装包覆盖升级
- 用户数据目录独立保存
- 数据库自动迁移
- 默认升级前自动备份数据库
- 预测结果默认软删除
- 设置中支持永久清理
